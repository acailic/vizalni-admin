## 1. Goal

Build a typed data transformation pipeline that loads CSV/JSON resources from data.gov.rs, parses them into typed observations, classifies columns as dimensions vs. measures, and provides transformation utilities ready for chart consumption in Feature 03.

## 3. Implementation Steps

1. Define observation types in `src/types/observation.ts`, then re-export them from `src/types/index.ts`.
2. Create `src/lib/data/loader.ts` with `loadResourceFromUrl`, `parseCsvContent`, and `parseJsonContent`.
3. Add BOM stripping, UTF-8/Windows-1250 handling, and Serbian numeric parsing (`1.234,56` â†’ `1234.56`).
4. Create `src/lib/data/classifier.ts` with heuristics for measure, temporal dimension, categorical dimension, and metadata columns.
5. Create `src/lib/data/transforms.ts` with `filterObservations`, `sortObservations`, `aggregateObservations`, `pivotObservations`, `computePercentages`, and `imputeMissing`.
6. Create `src/lib/data/index.ts` to expose the public pipeline API.
7. Extend `src/lib/api/datagov-client.ts` with `fetchResourceData(resourceUrl, format): Promise<ParsedDataset>`.
8. Add unit tests for loader, classifier, and transforms, covering malformed data and edge cases.

## 4. Translation Impact

## 6. Risks

| Risk | Mitigation |
|------|------------|
| CSV quality variability | Use robust parser defaults and return descriptive errors |
| Large files blocking UI | Use streaming/chunked parsing; defer worker offload if needed |
| Encoding misdetection | Try UTF-8 first, then Windows-1250 fallback |
| False classification of numeric IDs as measures | Treat classification as a hint; allow override later |
| Duplicate type definitions across repo | Keep new pipeline types isolated in `src/types/observation.ts` |
